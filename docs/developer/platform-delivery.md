# 平台交付与性能收尾指南

## 目标

阶段 F 不再优先做零散优化，而是把已经存在的能力整理成可部署、可解释、可继续演进的平台策略。

## 当前已落地的基础

- 长讨论页已使用增量加载和帖子流分页，而不是一次性拉全量帖子。
- 讨论列表、搜索结果、通知列表和后台列表统一走分页查询。
- 实时广播、输入提示、通知投递已经收口到统一 service 和事件入口。
- 主题模式与基础语言包已经进入注册中心和设置体系。

## 建议的性能策略

### 长讨论页

1. 默认保持窗口化加载，只在用户继续滚动时请求下一段帖子。
2. 任何新增帖子态字段优先走 Resource 字段扩展，不在页面层反复拼装。
3. 实时事件只广播增量 payload，避免重复下发整页讨论数据。

### 列表与搜索深分页

1. 现阶段继续使用稳定分页参数，但避免把单页大小无限放大。
2. 深分页问题优先通过筛选、排序和更明确的搜索语法缓解。
3. 如果后续搜索量继续增长，再评估 cursor 或 search-after 方案，而不是提前引入复杂协议。

### 实时广播降噪

1. 副作用事件统一在事务提交后触发。
2. 输入提示保持可开关，默认由 `advanced.realtime_typing_enabled` 控制。
3. 新增广播事件前，先判断是否能复用已有事件类型，避免把客户端状态推向细粒度噪音。

### 搜索索引刷新

1. 保持索引定义集中在搜索服务，不把索引刷新逻辑散落到业务 service。
2. 大批量重建优先走后台操作入口和命令，而不是请求链路内同步刷新。
3. 新增可搜索内容时，先确认帖子类型、资源字段和搜索过滤器是否已经声明完整。

## 平台交付清单

发布前至少检查以下事项：

1. `./.venv/bin/pytest` 全量通过。
2. `./.venv/bin/python manage.py test` 全量通过。
3. `cd frontend && npm test -- --runInBand` 全量通过。
4. 后台扩展中心可看到扩展依赖、生命周期、事件监听和资源字段。
5. 开发者文档页可直接说明扩展接入、资源字段、事件订阅和前端注入点。
6. 生产环境运行时检查能暴露缓存、队列、WebSocket 和搜索的关键风险。

## 暂不提前做的事

- 不提前引入外部插件市场。
- 不提前做复杂热插拔模块系统。
- 不为了“理论最优分页”重写现有全部列表接口。
- 不把国际化和主题系统扩成脱离论坛主场景的抽象平台。
